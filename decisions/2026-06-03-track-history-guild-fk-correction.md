# track_history guild FK should reference guilds.discordId, not guilds.id

**Date:** 2026-06-03  
**Status:** Accepted  
**Sentry:** LUCKY-3N (20 occurrences — FK constraint on `track_history_guildId_fkey`)

## Context

`TrackHistory.guildId` stores the Discord guild snowflake (passed as `queue.guild.id` throughout the codebase). The FK constraint was written to reference `guilds.id` — the CUID primary key auto-generated by Prisma.

When Guild records were created historically, `guilds.id` was set to the Discord snowflake by callers (pre-Prisma-CUID convention). After the schema adopted `@default(cuid())`, new Guild rows get auto-generated CUIDs, causing the FK to fail for any guild that joined after the convention change — their Discord snowflake in `track_history.guildId` no longer matches any `guilds.id`.

All existing `track_history` rows already hold Discord snowflakes as `guildId`. The data was always correct; the FK pointer was wrong.

## Decision

Change the FK constraint to reference `guilds.discordId` instead of `guilds.id`.

`guilds.discordId` is `@unique` and indexed. Cascade delete semantics are unchanged. Discord snowflakes are immutable. No data migration is required — existing rows already hold valid Discord snowflakes that match `guilds.discordId`.

## Alternatives considered

**Lookup at call time (Option D):** In `addTrackToHistory`, resolve the CUID by querying `prisma.guild.findUnique({ where: { discordId } })` then use the returned `id`. Rejected — patches the symptom, adds a DB roundtrip per track save, leaves the model misaligned.

**Set Guild.id = Discord snowflake at creation:** Explicitly pass `id: guild.id` when creating Guild rows to restore the old behaviour. Rejected — requires data migration of all existing CUID `guilds.id` rows and all other tables with FKs referencing `guilds.id`; high-risk.

**Silent handling (warnLog only):** Demote the error log and return false. Rejected — history is silently dropped for all new guilds; root cause unaddressed.

## Consequences

- **Positive:** LUCKY-3N stops firing. Track history is correctly saved for all guilds going forward.
- **Positive:** `addTrackToHistory` requires no code changes — `queue.guild.id` was already the correct value.
- **Neutral:** The migration is DROP + ADD CONSTRAINT only; non-destructive.
- **Negative:** None identified.

## Revisit when

- Discord changes the snowflake format or makes guild IDs mutable (essentially never).
- A future migration changes `guilds.discordId` to a non-unique column.
